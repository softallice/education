## 1. 성적 구성 (예시안)

아래 배점은 **조정 가능한 초안**이다. 최종 배점은 1주차 오리엔테이션에서 수강 인원·수준·강의 여건을 반영해 확정·공지한다.

| 항목 | 배점 | 설명 |
|------|------|------|
| 출석·참여 | 10 | 출석, 실습 시간 참여도, 질의응답·피어리뷰 기여 |
| 주차별 실습 과제 | 20 | 12주 강의 중 매주 부여되는 follow-along 실습 과제 (docpilot 진화 반영) |
| 블록 미니 과제 | 20 | 3개 블록 종료 시점 통합 미니 과제 — 블록1(5주, 클라우드 네이티브) 7점 / 블록2(8주, AI 연계) 7점 / 블록3(11주 Multi-Agent 종료, Agent) 6점 (합 20) |
| 몰입형 프로젝트 | 40 | P-학기 후반 4주간 팀 단위로 진행하는 캡스톤형 프로젝트 |
| 발표·문서화 | 10 | 최종 발표, 데모, README/아키텍처 문서 품질 |
| **합계** | **100** | |

> 배점 조정 시 원칙: **몰입형 프로젝트 + 주차별 과제 + 블록 미니 과제**의 합(실습 기반 평가)이 전체의 70% 이상을 유지한다. 이 과목은 "동작하는 결과물"을 최우선으로 평가한다.

### 1.1 성적 등급 기준 (참고)

- 절대 점수 기준: A (90~100), B (80~89), C (70~79), D (60~69), F (60 미만).
- **최종 등급은 학과 상대평가 규정(등급별 정원 비율)이 지배한다.** 절대 점수 기준은 안전장치(하한 게이트)로만 작동한다: 상대 순위가 높아도 절대 60점 미만이면 F가 될 수 있고, 절대 90점을 넘어도 상대 정원을 초과하면 A가 보장되지 않는다. (즉, 최종 성적은 상대평가가 결정하고 절대컷은 이를 뒤집지 못하되 하한 F 판정에는 우선한다.)

## 2. 평가 원칙

이 과목의 모든 평가는 다음 3가지 원칙을 기준으로 한다.

1. **동작하는 결과물 (Working Software)**
   - 슬라이드·설명만으로는 점수를 받을 수 없다. 실제로 빌드·실행·배포되는 산출물이 있어야 한다.
   - "내 노트북에서만 됨"은 감점 요인. 컨테이너/K8s로 **재현 가능한 실행**이 기본.

2. **설명 가능성 (Explainability)**
   - 코드를 붙여넣기만 하고 설명하지 못하면 감점. 왜 이 구조·이 API·이 도구를 선택했는지 설명할 수 있어야 한다.
   - 발표·데모 중 질의응답으로 개별 기여와 이해도를 확인한다.

3. **클라우드 네이티브 준수 (Cloud-Native Compliance)**
   - 12-Factor 원칙(설정 외부화, 무상태화, 로그 표준출력 등) 준수.
   - 컨테이너화, 헬스체크(probe), 설정·시크릿 분리, 선언형 배포(K8s manifest)를 지킨다.
   - 하드코딩된 시크릿·API 키는 보안 원칙 위반으로 **발견 1건당 해당 평가 항목에서 -2점**(주차 과제 배점 또는 몰입형 '클라우드 네이티브' 항목)을 부과하고, 즉시 정정·키 회전을 요구한다. 감점은 해당 항목 배점을 상한으로 한다.

## 3. 주차별 과제 제출 방식

모든 주차 과제와 프로젝트는 **GitHub 기반**으로 제출한다.

### 3.1 저장소 규칙

- 학생별 개인 저장소: `docpilot-<학번>` (예: `docpilot-201912345`). 매주 같은 저장소가 자란다(러닝 프로젝트).
- 저장소는 **비공개(private)**로 두고 담당 교수/조교 계정을 collaborator로 초대한다.
- 루트에 `README.md`(실행 방법), `.env.example`(키 목록, 실제 키는 커밋 금지)를 유지한다.

### 3.2 제출 흐름 (매주)

```bash
# 1. 이번 주 작업 브랜치 생성 (무엇을: 주차별 작업 격리 / 왜: 리뷰·롤백 용이)
git checkout -b week03-compose

# 2. 실습 진행 후 커밋 (커밋 메시지에 주차·핵심 명시)
git add .
git commit -m "feat(week03): docker-compose로 web+postgres 구성"

# 3. 원격 푸시 후 PR 생성
git push -u origin week03-compose
gh pr create --title "Week03 제출" --body "docpilot: compose 멀티컨테이너"
```

- **확인**: PR이 열리고, PR 본문에 (1) 실행 스크린샷/로그, (2) 기대 출력과 실제 출력, (3) 트러블슈팅 요약이 포함되어야 한다.

### 3.3 제출물 형식

| 구성 | 필수 | 내용 |
|------|------|------|
| GitHub PR | O | 주차별 브랜치 → main PR. 마감 시각의 최신 커밋으로 채점 |
| 데모 | O | 실습 시간 내 3분 데모 또는 화면 녹화 링크 |
| 실행 로그/스크린샷 | O | 기대 출력 도달 증빙 (PR 본문 첨부) |
| 회고 3줄 | 권장 | 배운 점 / 막힌 점 / 다음에 시도할 것 |

- **마감**: 원칙적으로 다음 강의 전날 23:59. **지각 감점**은 다음으로 고정한다: 마감 후 24시간 이내 해당 과제 배점의 **-20%**, 24~72시간 **-50%**, 72시간 초과 **0점**(미제출 처리).

## 4. 팀 구성 가이드 (몰입형 프로젝트)

몰입형 프로젝트는 **팀 단위**로 진행한다. 개인 러닝 프로젝트(docpilot)로 다진 역량을 팀 협업으로 확장한다.

### 4.1 팀 규모

- **3~4명으로 구성한다** (최소 3명, 최대 4명).
- 2명 이하는 산출물 요구사항(멀티에이전트 등)을 4주 안에 소화하기 어렵고, 5명 이상은 기여도 편차·무임승차 위험이 커진다.
- **3인 팀 산출물 경감**: 필수 요소 중 멀티에이전트는 '단일 Agent + MCP 도구 노출'로 대체 가능하다(멀티에이전트 구현 시 가점). 문서 세트는 그대로 유지하되 데모 영상은 라이브 데모로 갈음할 수 있다.

### 4.2 역할 분담 예시

팀당 아래 4개 역할을 나눠 맡는다(3인 팀은 겸임). 역할은 **주 소유자**를 정하되 전원이 전체 흐름을 이해해야 한다(설명 가능성 원칙).

| 역할 | 주 책임 | 대표 산출물 |
|------|---------|------------|
| 클라우드/배포 담당 | 컨테이너화, K8s manifest, CI, 배포·롤백, 관측성 | Dockerfile, k8s/*.yaml, GitHub Actions |
| AI/모델 담당 | LLM·임베딩·멀티모달 연동, 프롬프트, RAG 파이프라인 | `/chat`, `/ask`, 벡터DB 인덱싱 |
| Agent/백엔드 담당 | function calling, ReAct, MCP, 멀티에이전트 오케스트레이션, API | `/agent`, tool 정의, MCP 서버 |
| PM/문서 담당 | 일정·이슈 관리, 데모 시나리오, README·아키텍처 문서, 발표 | 프로젝트 보드, 아키텍처 다이어그램, 발표자료 |

### 4.3 팀 구성 시점

- **12주차 킥오프**에서 팀을 확정한다(주차별 커리큘럼 week12 참조).
- 12주차 전까지 개인별 docpilot 진척도·관심 주제를 공유해 팀 매칭에 활용한다.
- 팀 확정 후 즉시 팀 저장소(`docpilot-team-<번호>`)를 개설하고 역할·주제 초안을 등록한다.

### 4.4 협업 규칙

- **브랜치 전략**: `main`(항상 배포 가능) + `feature/<역할-기능>` 브랜치 → PR → 리뷰 1인 이상 승인 후 머지. `main` 직접 push 금지.
- **이슈 관리**: GitHub Issues + Projects(칸반: Todo / Doing / Review / Done). 모든 작업은 이슈로 등록하고 커밋/PR에서 `#이슈번호`로 연결.
- **역할 로테이션**: 4주 중 최소 1회는 페어링/리뷰를 통해 타 역할 영역을 경험한다(예: AI 담당이 배포 PR을 리뷰). 무임승차 방지 및 상호 학습 목적.
- **커뮤니케이션**: 주 2회 스탠드업(진행/막힘/다음), 기록은 이슈 코멘트 또는 팀 위키에 남긴다.
- **기여도 측정(M1, 편향 보정)**: 커밋 수 단독으로 평가하지 않는다(PM·문서 담당이 불리해지는 것을 방지). 개인 기여는 **역할별 산출물 기준**(4.2 표의 대표 산출물: 예) PM/문서 담당은 프로젝트 보드 운영·아키텍처 다이어그램·발표자료, 클라우드 담당은 Dockerfile·k8s manifest·CI)의 완성도로 1차 평가하고, 커밋·PR·이슈는 **보조 지표**로만 사용한다.
- **피어 평가 규정**: 상호 피어 평가는 **익명**으로 수집한다. 평가자별 점수 분포를 **정규화**(평균·표준편차 보정)해 관대/엄격 편향을 제거한 뒤 반영하며, 자기 자신은 평가 대상에서 제외한다.

## 5. 몰입형 프로젝트 (4주)

P-학기 후반 4주간 진행하는 캡스톤형 프로젝트. 12주 강의에서 배운 세 축(클라우드 네이티브 / AI 연계 / Agent)을 하나의 서비스로 통합한다.

### 5.1 요구사항 — 필수 요소

프로젝트는 아래 4가지를 **모두 포함**해야 한다(하나라도 빠지면 요구사항 미충족).

1. **AI API 연계**: OpenAI / Gemini / Hugging Face 중 1개 이상을 실제로 호출. 스트리밍·프롬프트 설계 포함. (무료/저비용 대안: HF 무료 티어, 로컬 Ollama 허용)
2. **Agent 요소**: 단일 Agent(function/tool calling, ReAct) 이상. 가능하면 MCP 도구 노출 또는 멀티에이전트 협업까지 확장.
3. **컨테이너화 / K8s 배포**: Docker 이미지화 + kind/minikube에 Deployment·Service 배포. ConfigMap/Secret로 설정 분리, probe 구성.
4. **문서화**: README(실행법), 아키텍처 다이어그램, API 명세, 데모 시나리오. 재현 가능해야 한다.

### 5.2 4주 주차별 진행표

| 주 | 목표 | 세부 활동 | 산출물 |
|----|------|-----------|--------|
| P1 (13주차) | 킥오프·설계 | 주제 확정, 요구사항 정의, 아키텍처 설계, 역할·이슈 세팅, 기술 스파이크 | PRD, 아키텍처 문서, 프로젝트 보드, 스켈레톤 저장소 |
| P2 (14주차) | 핵심 기능 구현 | AI API 연계 + 기본 Agent 동작, 로컬/compose 실행 | 동작하는 백엔드, `/chat`·`/agent` 최소 기능, 컨테이너 이미지 |
| P3 (15주차) | 통합·배포 | Agent 고도화(MCP/멀티에이전트), K8s 배포, ConfigMap/Secret/probe, CI | K8s 배포본, CI 파이프라인, 통합 테스트 |
| P4 (16주차) | 마무리·발표 | 관측성·안정화, 문서 완성, 데모 리허설, 최종 발표 | 최종 배포본, 문서 세트, 발표자료, 데모 영상 |

> P-학기 정확한 날짜는 [학사 일정](01-학사일정.md)을 따른다. 위 13~16주차는 12주 강의 이후 4주 몰입 기간을 가리키는 상대 표기다.

### 5.3 산출물 (최종 제출)

- **소스 코드**: 팀 GitHub 저장소(`main` 배포 가능 상태).
- **컨테이너 이미지**: 레지스트리 푸시 또는 빌드 스크립트로 재현 가능.
- **K8s manifest**: `k8s/` 디렉터리(Deployment, Service, ConfigMap, Secret 예시, Ingress/HPA 선택).
- **문서 세트**: `README.md`, `docs/architecture.md`(다이어그램 포함), `docs/api.md`, `docs/demo.md`.
- **발표자료 + 데모 영상**: 10분 발표 + 5분 데모(또는 녹화 링크).

### 5.4 평가 루브릭

몰입형 프로젝트 40점을 아래 6개 기준으로 분배한다. 각 기준은 **4수준(우수/양호/보통/미흡)**으로 채점하고, 각 수준의 점수는 관찰 가능한 기술어로 판정한다. 우수 열의 합은 40점으로 고정된다.

| 평가 항목 | 배점 | 우수 | 양호 | 보통 | 미흡 |
|-----------|------|------|------|------|------|
| 기능 완성도(동작성) | 12 | 12 · 요구 기능 전체가 명세대로 동작, 핵심 플로우 무오류 | 9 · 핵심 기능 동작, 부가 기능 일부 미흡 | 6 · 핵심 기능 중 일부만 동작 | 3 · 대부분 미완성/실행 불가 |
| AI 연계 | 5 | 5 · 외부/로컬 AI API 실호출 + 스트리밍·프롬프트 설계 + 예외 처리 | 4 · API 실호출 동작, 설계 요소 일부 결여 | 2 · 호출은 되나 단발성/하드코딩 | 1 · AI 연계 사실상 없음 |
| Agent·MCP·멀티에이전트 | 5 | 5 · MCP 도구 노출 또는 멀티에이전트 협업이 동작 | 4 · 단일 Agent(tool/function calling, ReAct)가 견고 | 2 · Agent 흉내, 도구 호출 불안정 | 1 · Agent 부재 |
| 클라우드 네이티브 | 8 | 8 · 컨테이너화+K8s 배포+Config/Secret 분리+probe+CI 충족 | 6 · K8s 배포 동작, 일부 요소 누락 | 4 · 컨테이너화/compose 수준에 그침 | 2 · 배포 미구성 |
| 문서화·재현성 | 5 | 5 · 문서만으로 제3자 재현 성공, 아키텍처 명확 | 4 · 재현 가능하나 일부 수동 보완 필요 | 2 · 문서 부실, 재현에 저자 도움 필요 | 1 · 재현 불가 |
| 발표·데모·Q&A | 5 | 5 · 논리적 발표+라이브 데모 성공+Q&A 명확 | 4 · 발표·데모 양호, Q&A 일부 미흡 | 2 · 데모 부분 성공, 설명 약함 | 1 · 데모 실패/설명 불가 |
| **합계** | **40** | **40** | — | — | — |

> **중복 처벌 제거(M3)**: '기능 완성도(동작성)'는 **기능이 명세대로 동작하는지**만 평가하며 배포 성공 여부와 독립이다. 배포 품질은 '클라우드 네이티브'에서, 재현 가능성은 '문서화·재현성'에서만 평가한다. 배포가 실패해도 기능이 로컬/compose에서 명세대로 동작하면 기능 완성도는 감점하지 않는다(하나의 결함을 세 항목에서 3중 감점하지 않는다).

> **발표·데모·Q&A(M7) 5점 내역**: 발표·데모 전달 3점 + **설명 가능성 Q&A 2점**. Q&A는 팀 전체 질문 1개 이상과 **개별 팀원 지목 질문 1개 이상**을 반드시 포함한다. 지목된 팀원이 자기 담당 부분의 구조·API·도구 선택 이유를 설명하면 pass. Q&A 하위 4수준: 2점(지목 질문에 근거 있는 답변) / 1.5점(대체로 설명) / 1점(부분 설명) / 0.5점(설명 불가).

#### 5.4.1 조작적 정의 (pass/fail 체크리스트)

"안정적으로 동작", "목적에 맞게 설계" 같은 주관적 표현을 아래 **측정 가능한 pass/fail 판정**으로 환원한다. 각 항목은 pass/fail로 기록하며 루브릭 수준 판정의 근거가 된다.

- 기능 완성도: 핵심 API에 **동일 요청 20회 연속 무오류**(5xx 0건) → pass
- 기능 완성도: 명세된 사용자 플로우를 **엔드투엔드 1회 이상 재현 성공** → pass
- AI 연계: 실제 외부/로컬 AI 엔드포인트로부터 **응답 수신 로그가 존재** → pass
- Agent·MCP·멀티에이전트: tool/function call이 **실제 도구 실행 결과를 반영**(모의 응답 아님) → pass
- 클라우드 네이티브: K8s에서 **Pod Running + readiness probe 통과** → pass
- 클라우드 네이티브: 리포지터리 스캔 결과 **Secret 하드코딩 0건** → pass
- 문서화·재현성: 제3자가 README만으로 **로컬 또는 배포 기동 성공** → pass

> **개인 점수 조정(M1)**: 팀 점수 대비 개인 **±20%** 범위에서 조정한다. 조정 근거의 우선순위는 (1) **역할별 산출물 기준 달성도**(주 근거, 4.2 표), (2) **익명·정규화 피어 평가**, (3) 보조 지표로서 커밋/PR/이슈다. **커밋 수는 단독 근거가 될 수 없다.**

### 5.5 주제 예시

아래는 요구사항(AI + Agent + 컨테이너/K8s + 문서화)을 자연스럽게 충족하는 예시다. 그대로 써도 되고 변형·신규 주제도 가능(사전 승인).

- **문서 QA Agent**: 사내/강의 문서를 올리면 RAG로 근거를 찾아 답하고, 필요 시 도구(검색·계산·DB 조회)를 호출하는 Agent. → docpilot의 직계 확장.
- **멀티에이전트 리서치 어시스턴트**: researcher(자료 수집) + writer(보고서 작성) + reviewer(검증) 협업으로 리서치 보고서를 생성.
- **코드/데이터 분석 Agent**: 업로드된 코드·CSV를 분석하고, 도구(파이썬 실행·쿼리)를 호출해 결과·시각화를 반환.
- **멀티모달 상담봇**: 텍스트+이미지(또는 음성) 입력을 이해하고, 지식베이스 검색과 도구 호출로 응대하는 상담 Agent.

### 5.6 docpilot 확장 경로 (권장)

12주 내내 키운 러닝 프로젝트 **docpilot을 포크/확장**해 팀 프로젝트로 발전시키는 것을 권장한다. 처음부터 새로 만드는 것보다 학습 성과가 크고, 요구사항 충족도 쉽다.

- **포크 시작**: 팀원 중 가장 완성도 높은 docpilot을 팀 저장소로 포크 → 공동 개발.
- **확장 방향 예시**:
  - RAG 고도화(하이브리드 검색, re-ranking), 멀티에이전트(researcher+writer)로 보고서 생성.
  - MCP 서버로 docpilot 도구를 외부(다른 Agent/클라이언트)에 노출.
  - 멀티모달 입력(이미지/음성) 추가, 관측성(로그·메트릭) 대시보드.
  - K8s 운영형 배포(HPA, Ingress, 롤아웃/롤백)로 완성도 상향.
- **주의**: 확장이라도 **팀 고유의 새 가치**(새 도메인·새 기능·새 통합)를 최소 하나 이상 추가해야 한다. 단순 재배포는 인정하지 않는다.
