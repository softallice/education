개념 ② / ReAct (Reasoning + Acting)

Thought세율을 먼저 문서에서 찾아야겠다.

Function calling = ReAct의 구조화

모델이 자연어로 Action: calculator(...)를 쓰는 대신, API가 정한 tool_calls 필드에 함수 이름 + JSON 인자를 채워 돌려준다.

우리는 그걸 실행해 결과를 role:"tool" 메시지로 다시 넣어준다. 이 왕복이 곧 Action → Observation이다.
개념 ③ / Function·Tool calling 원리 (4단계)

모델은 고르기만, 실행은 우리 몫

1스키마 전달

도구의 이름·설명·파라미터(JSON Schema)를 tools 인자로 모델에 넘긴다.

2모델이 선택

필요하다 판단하면 텍스트 대신 tool_calls(함수명 + JSON 인자)를 반환.

3우리가 실행

모델은 실행 못 한다. 우리 코드가 인자를 파싱해 진짜 파이썬 함수를 호출.

4결과 반환

결과를 role:"tool" 메시지로 추가하고 다시 모델 호출 → 다음 행동/최종 답.

핵심: 모델은 "무엇을 호출할지"만 정한다. 실제 실행과 안전은 전부 우리 몫이다. 이 원리가 week10의 가드레일로 이어진다.

agent/tools.py — 도구 1: 안전한 계산기
import ast, operator
# eval()은 위험 → AST를 walk하며 산술 연산자만 허용
_ALLOWED_OPERATORS = {
    ast.Add: operator.add, ast.Sub: operator.sub,
    ast.Mult: operator.mul, ast.Div: operator.truediv,
    ast.Pow: operator.pow, ast.Mod: operator.mod,
    ast.USub: operator.neg, ast.UAdd: operator.pos,
}


def _eval_node(node: ast.AST) -> float:
    if isinstance(node, ast.Constant):
        return float(node.value)
    if isinstance(node, ast.BinOp):
        op = _ALLOWED_OPERATORS[type(node.op)]
        return op(_eval_node(node.left),
                  _eval_node(node.right))
    if isinstance(node, ast.UnaryOp):
        op = _ALLOWED_OPERATORS[type(node.op)]
        return op(_eval_node(node.operand))
    raise ValueError("허용되지 않은 수식입니다.")


def calculator(expression: str) -> float:
    """산술 수식을 계산. 예: '50000 * 1.1'."""
    return _eval_node(ast.parse(expression, mode="eval").body)
agent/tools.py — 도구 2: 현재 시각
import datetime as dt
from zoneinfo import ZoneInfo


def current_time(timezone: str = "Asia/Seoul") -> str:
    """지정 타임존의 현재 시각을 ISO 8601로 반환."""
    try:
        tz = ZoneInfo(timezone)
    except Exception:
        tz = ZoneInfo("Asia/Seoul")
    return dt.datetime.now(tz).isoformat(
        timespec="seconds")

도구는 순수 파이썬 함수다. 계산기는 eval을 피하고 AST로 입력을 검증한다 — week10 가드레일의 기본기.

왜 스키마와 함수를 나누나

스키마는 "모델에게 도구를 설명하는 문서", 레지스트리는 "실제 실행 대상". 모델은 스키마만 보고 이름·인자를 정하고, 우리는 레지스트리로 진짜 실행한다.

raw_agent.py — 루프의 뼈대

agent/raw_agent.py — 설정 + 모델 호출
import json, os
from openai import OpenAI
from agent.tools import TOOL_REGISTRY, TOOL_SCHEMAS

MODEL = os.getenv("OPENAI_MODEL", "gpt-4o-mini")
MAX_STEPS = 6  # 무한 루프 방지: 왕복 상한
SYSTEM_PROMPT = (
    "너는 docpilot의 assistant다. 필요하면 도구를 써라. "
    "문서 근거가 필요하면 rag_search를 먼저, "
    "계산은 calculator를. 근거 없이 지어내지 마라.")
client = OpenAI()  # OPENAI_API_KEY 자동 로드


def run_agent(user_query: str, verbose=True) -> dict:
    messages = [{"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user", "content": user_query}]
    trace = []  # 추론 과정 관찰용 로그
    for step in range(1, MAX_STEPS + 1):
        # (1) 도구 사용 여부는 모델이 결정
        response = client.chat.completions.create(
            model=MODEL, messages=messages,
            tools=TOOL_SCHEMAS, tool_choice="auto")
        msg = response.choices[0].message
        # (2) 도구 호출이 없으면 최종 답 → 종료
        if not msg.tool_calls:
            return {"answer": msg.content,
                    "trace": trace, "steps": step}
        # ... (다음 슬라이드: 왕복) ...
tool_choice="auto"

도구를 쓸지 말지는 모델이 판단한다. 도구 호출이 없다는 것은 "더 할 일이 없다 = 최종 답"이라는 신호.

MAX_STEPS로 왕복 횟수에 상한을 둬 무한 루프를 막는다. trace가 곧 추론 과정의 기록.

이것이 이번 주의 핵심. 프레임워크가 뒤에서 무엇을 하는지 직접 보기 위해 손으로 짠다.

assistant ↔ role:tool 왕복

모델의 tool_calls를 그대로 대화에 추가(규격 준수)한 뒤, 실행 결과를 tool_call_id로 짝지어 role:"tool"로 되돌린다.

도구 오류도 삼키지 않고 모델에게 문자열로 알려 회복시킨다. 직렬화는 json.dumps(..., default=str)로 JSON 불가 타입까지 안전하게.

agent/api.py — /agent 엔드포인트

agent/api.py
from fastapi import APIRouter
from pydantic import BaseModel, Field
from agent.raw_agent import run_agent

router = APIRouter(prefix="/agent", tags=["agent"])


class AgentRequest(BaseModel):
    query: str = Field(..., min_length=1,
        max_length=2000, description="사용자 질의")


class AgentResponse(BaseModel):
    answer: str
    steps: int
    trace: list[dict]


@router.post("", response_model=AgentResponse)
def ask_agent(req: AgentRequest) -> AgentResponse:
    out = run_agent(req.query, verbose=False)
    return AgentResponse(answer=out["answer"],
        steps=out["steps"], trace=out["trace"])
응답에 trace를 그대로 노출

answer + steps + trace를 반환한다. 어떤 도구를 어떤 인자로 호출했는지가 API 응답에 그대로 드러난다.

이것이 디버깅과 신뢰성 확보의 출발점이다. Pydantic Field로 입력 길이를 검증(1~2000자).

agent/graph_agent.py — create_react_agent
# pip install "langgraph>=0.2" "langchain-openai>=0.2"
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from agent import tools as t


# 기존 순수 함수를 도구로 감싼다. docstring = 도구 설명.
@tool
def calculator(expression: str) -> float:
    """산술 수식을 계산한다. 예: '50000 * 1.1'."""
    return t.calculator(expression)


@tool
def rag_search(query: str, top_k: int = 3) -> list:
    """docpilot 문서 지식베이스를 검색(week08 RAG)."""
    return t.rag_search(query, top_k=top_k)


llm = ChatOpenAI(model=MODEL, temperature=0)
graph_agent = create_react_agent(
    llm, [calculator, current_time, rag_search])

필수 경로는 2~3부에서 끝난다. 이 4부는 프레임워크 비교용 심화이며 건너뛰어도 무방하다.

raw vs LangGraph

raw는 루프·에러·중단을 전부 우리가 통제(학습·디버깅 최적). LangGraph는 상태·체크포인트·스트리밍을 제공해 복잡한 그래프(week11 Multi-Agent)로 확장하기 좋다. 원리는 동일하다.

pretty_print()가 Human → Ai(tool_calls) → Tool → Ai(최종답) 흐름을 보여준다 — raw의 trace와 같은 사이클.

도구·데이터 연동과 MCP

이번 주 도구들은 아직 장난감이다. 다음 주엔 외부 REST API·PostgreSQL·파일 시스템에 실제로 연결하고, 타임아웃·재시도·입력검증 가드레일을 두른다. 그리고 도구들을 MCP(Model Context Protocol) 서버로 노출해 어떤 MCP 클라이언트든 호출하게 만든다.
