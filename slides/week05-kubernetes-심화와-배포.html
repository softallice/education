개념 ① / 설정 외부화 (12-Factor)

"설정은 환경에 저장하라"

코드/이미지는 환경에 무관하게 동일해야 하고, 환경별로 달라지는 값(DB 주소, 로그레벨, 자격증명)은 밖에서 주입한다. week04에선 YAML에 DATABASE_URL·비밀번호를 박아 넣어 이 원칙을 어겼다 — 이번 주에 바로잡는다.

ConfigMap

비밀이 아닌 설정값. 예: APP_ENV, LOG_LEVEL. 평문으로 저장·조회된다.

Secret

민감정보. 예: DB 비밀번호, API 키. 값이 base64 인코딩되어 저장된다.

⚠️ Secret은 암호화가 아니라 인코딩이다

base64는 누구나 되돌릴 수 있다. 진짜 운영은 저장소 암호화·Sealed/External Secrets·KMS를 쓴다. 그리고 secret.yaml은 git에 커밋하지 않는다(.gitignore).

개념 ② / 헬스체크: probe

K8s가 컨테이너 상태를 묻는 방식

probe질문실패하면docpilot 설정
liveness"이 컨테이너 살아있나(교착 아닌가)?"컨테이너를 재시작GET /health
readiness"이제 트래픽 받아도 되나?"Service 대상에서 제외(트래픽 차단)GET /health
startup"느린 초기화가 끝났나?"초기화 대기 (이번 주 생략)—

week04의 간헐 실패는 readiness가 해결한다. 준비 안 된 새 Pod는 Service가 트래픽을 안 보낸다 → 무중단 롤아웃의 핵심.

개념 ③④ / 리소스 · 오토스케일링

requests가 있어야 HPA가 돈다

requests / limits

requests: 스케줄러가 최소 보장하는 양. HPA 사용률 계산 기준.
limits: 상한. CPU 초과는 throttling, 메모리 초과는 OOMKill.

requests가 없으면 HPA는 사용률을 <unknown>으로 표시한다. "CPU 50%" 목표를 계산할 기준이 없기 때문.

HPA (HorizontalPodAutoscaler)

지표(CPU 등)를 보고 Deployment의 replicas를 자동 조정한다. 지표 수집을 위해 metrics-server가 필요하다 — kind엔 기본 미설치라 직접 설치·패치한다.

개념 ⑤⑥⑦ / 노출 · 배포 · 통합

바깥 세상과 연결하기

⑤ Ingress

Service(ClusterIP)는 클러스터 내부 전용. Ingress는 바깥 HTTP(도메인/경로)를 내부 Service로 라우팅하는 L7 진입점. 규칙을 처리하려면 Ingress Controller(ingress-nginx)가 필요.

⑥ 롤아웃 / 롤백

이미지·설정이 바뀌면 RollingUpdate로 Pod를 점진 교체. maxUnavailable/maxSurge로 가용성 통제, 문제가 생기면 이전 리비전으로 롤백.

⑦ CI / CD

CI: push마다 자동 빌드·이미지 생성(오늘 GitHub Actions). CD: 만든 이미지를 자동 배포(개념만 — kubeconfig 시크릿/GitOps 필요).

순서: 1부 설정 외부화 → 2부 probe·리소스 → 3부 HPA → 4부 Ingress → 5부 롤아웃/롤백 → 6부 CI/CD. 모든 명령은 docpilot/ 프로젝트 루트에서.

1부 · 1단계

ConfigMap — 비밀 아닌 설정값

k8s/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: docpilot-config
data:
  APP_ENV: "k8s-prod"
  LOG_LEVEL: "info"

data에 평문 key/value를 적는다. 앱은 이 값을 환경변수로 받게 되고, 이미지를 다시 빌드하지 않고도 환경별로 바꿀 수 있다.

여기 값은 비밀이 아니므로 git에 커밋해도 된다. 민감정보는 다음 단계 Secret으로 분리한다.

1부 · 2단계 · 실행 확인

Secret — 민감정보 (그리고 base64 체감)

k8s/secret.yaml ← git 커밋 금지
apiVersion: v1
kind: Secret
metadata:
  name: docpilot-db-secret
type: Opaque
stringData:                 # 평문 → K8s가 저장 시 base64
  POSTGRES_USER: "docpilot"
  POSTGRES_PASSWORD: "docpilot"
  POSTGRES_DB: "docpilot"
  DATABASE_URL: "postgresql+psycopg://docpilot:docpilot@docpilot-db:5432/docpilot"
bash — 무시목록 등록 · 적용
echo "k8s/secret.yaml" >> .gitignore
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/secret.yaml
# base64는 누구나 복원 가능함을 직접 확인
kubectl get secret docpilot-db-secret \
  -o jsonpath='{.data.POSTGRES_PASSWORD}' | base64 -d; echo
# → docpilot

stringData를 쓰면 직접 인코딩할 필요 없이 평문으로 적어도 K8s가 저장 시 base64로 바꿔 준다.

1부 · 3단계

Deployment를 외부화된 설정으로 (week05 개정판)

k8s/deployment.yaml — 외부화 + 무중단 전략
spec:
  replicas: 2
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 교체 중 최대 1개 초과
      maxUnavailable: 0  # 사용불가 0 → 무중단
  # ... template.spec.containers[0] ...
      image: docpilot:0.3.0
      imagePullPolicy: IfNotPresent
      envFrom:
        - configMapRef:
            name: docpilot-config   # APP_ENV/LOG_LEVEL
      env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: docpilot-db-secret
              key: DATABASE_URL
같은 컨테이너 — resources · probe
      resources:
        requests:
          cpu: "100m"       # HPA 계산 기준
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
      livenessProbe:
        httpGet:
          path: /health
          port: 8000
        initialDelaySeconds: 5
        periodSeconds: 10
      readinessProbe:
        httpGet:
          path: /health
          port: 8000
        initialDelaySeconds: 3
        periodSeconds: 5

핵심: 하드코딩 env가 envFrom(ConfigMap)과 secretKeyRef(Secret)로 바뀌었다. probe·resources·롤아웃 전략도 함께 들어간다.

1부 · 4단계 · 실행 확인

PostgreSQL도 Secret에서 자격증명 주입

k8s/postgres.yaml — env 블록만 교체
env:
  - name: POSTGRES_USER
    valueFrom:
      secretKeyRef:
        name: docpilot-db-secret
        key: POSTGRES_USER
  - name: POSTGRES_PASSWORD
    valueFrom:
      secretKeyRef:
        name: docpilot-db-secret
        key: POSTGRES_PASSWORD
  - name: POSTGRES_DB
    valueFrom:
      secretKeyRef: { name: docpilot-db-secret, key: POSTGRES_DB }
bash — 적용 · 주입 검증
kubectl apply -f k8s/postgres.yaml
kubectl apply -f k8s/deployment.yaml
kubectl rollout status deployment/docpilot
POD=$(kubectl get pod -l app=docpilot \
  -o jsonpath='{.items[0].metadata.name}')
kubectl exec "$POD" -- printenv APP_ENV DATABASE_URL
기대 출력 
APP_ENV=k8s-prod
DATABASE_URL=postgresql+psycopg://docpilot:docpilot@docpilot-db:5432/docpilot

APP_ENV=ConfigMap, DATABASE_URL=Secret에서 주입됐으면 외부화 성공.

3부 · 8단계 · 실행 확인

HPA — CPU 50% 목표로 자동 스케일

k8s/hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: docpilot
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: docpilot
  minReplicas: 2
  maxReplicas: 6
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 50
bash — 적용·확인
kubectl apply -f k8s/hpa.yaml
kubectl get hpa docpilot
확인

TARGETS가 <unknown>이 아니라 2%/50% 같은 실제 값이면 정상. <unknown>이면 metrics-server 대기 또는 requests 누락.

4부 · 11·12단계 · 실행 확인

Ingress 작성 → 도메인으로 접속

k8s/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: docpilot
spec:
  ingressClassName: nginx
  rules:
    - host: docpilot.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: docpilot
                port: { number: 80 }
bash — 적용 · 도메인 등록 · 접속
kubectl apply -f k8s/ingress.yaml
kubectl get ingress docpilot   # ADDRESS 대기
# hosts에 도메인 등록 (한 번만)
echo "127.0.0.1 docpilot.local" | \
  sudo tee -a /etc/hosts
curl -s http://docpilot.local/health  # {"status":"ok"}
# hosts 수정이 어려우면 Host 헤더로 대체
curl -s -H "Host: docpilot.local" http://localhost/health

이제 port-forward 없이 도메인으로 접속된다. Ingress → Service → Pod 경로가 완성됐다. (week04에서 노드 80→호스트 80 매핑을 해 뒀기에 가능.)

문제 발생 가정 → 롤백

kubectl rollout undo deployment/docpilot
