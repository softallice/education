클라우드 네이티브,

클라우드 네이티브의 감을 잡고, 러닝 프로젝트 docpilot의 첫 코드를 로컬에서 실행해 GitHub에 올린다.

아무것도 없음 → FastAPI 로컬 웹앱 GET / · /health
01 / 개요

이번 주가 끝나면 할 수 있는 것

  • 1IaaS / PaaS / SaaS의 책임 경계를 한 문장으로 설명한다. 온프렘과 클라우드의 비용·운영 구조 차이를 말한다.
  • 2클라우드 네이티브의 핵심 특성과 12-Factor App 원칙을 예를 들어 설명한다.
  • 3모놀리스와 대비해 마이크로서비스가 왜 등장했는지 이유를 제시한다.
  • 4Python 가상환경을 만들고 FastAPI 앱을 uvicorn으로 실행한다.
  • 5git init → 첫 커밋 → GitHub 푸시까지 완주한다.
개념 ① / 클라우드 컴퓨팅

누가 어디까지 관리하는가

INFRASTRUCTURE

IaaS

제공자: 서버·네트워크·스토리지·가상화 나: OS·런타임·앱·데이터 AWS EC2 · GCP Compute Engine
PLATFORM

PaaS

제공자: 위 + OS·런타임 나: 앱 코드·데이터 Heroku · Cloud Run · App Engine
SOFTWARE

SaaS

제공자: 전부 나: 설정·데이터 입력만 Gmail · Notion · Slack

위로 갈수록 신경 쓸 게 줄지만 통제권도 줄어든다. docpilot은 IaaS/PaaS 위에 직접 올려 운영하는 앱을 만드는 과정이다.

개념 ② / 인프라 모델

온프레미스 vs 클라우드

구분온프레미스 (On-Prem)클라우드
초기 비용서버 구매 — CapEx, 큰 선투자없음 — OpEx, 쓴 만큼 과금
확장장비 추가 = 수주~수개월수 분 내 스케일
운영 책임전기·냉방·하드웨어까지 전부물리 계층은 제공자
적합규제·초저지연·고정 부하가변 부하·빠른 실험
개념 ③ / 클라우드 네이티브

클라우드에 올린 앱이 아니다

클라우드 네이티브 = 클라우드의 탄력성·자동화를 전제로 설계·운영하는 방식.

컨테이너로 패키징 · 선언적으로 배포
자동 확장 · 자가 치유되도록 설계

CNCF는 컨테이너 · 마이크로서비스 · 불변 인프라 · 선언적 API를 대표 특성으로 꼽는다. 이 강의의 블록 1(1~5주)이 바로 이걸 다룬다.

개념 ⑤ / 아키텍처

모놀리스 → 마이크로서비스

모놀리스

하나의 배포 단위. 시작은 단순하지만, 커지면 배포·확장이 통째로 묶인다.

마이크로서비스

기능별로 독립 배포·확장되는 작은 서비스들의 모음. 유연함을 얻는 대신 네트워크·운영 복잡도가 커진다.

작게 시작해서 필요할 때 쪼갠다

docpilot은 단일 서비스로 시작해, 필요할 때 DB·벡터DB·에이전트 등 백엔드를 붙여 나간다. 이게 실무 감각.

아래 9단계를 순서대로 그대로 따라 친다. 명령·코드마다 "무엇을/왜" 한 줄 설명과 실행 후 확인이 붙는다.

1부 · 1~3단계

스캐폴드 · 가상환경 · 의존성 고정

bash — 스캐폴드 & venv
# 1. 작업 디렉터리
mkdir -p ~/projects/docpilot
cd ~/projects/docpilot

# 2. 가상환경 (12-Factor #2)
python3 -m venv .venv
source .venv/bin/activate

# 3. 의존성 설치
pip install "fastapi==0.115.*" "uvicorn[standard]==0.32.*"
requirements.txt
fastapi==0.115.6
uvicorn[standard]==0.32.1
확인

프롬프트에 (.venv) 표시 · python -c "import fastapi, uvicorn; print('ok')" → ok

버전을 파일로 고정하면 팀 전체가 같은 환경 — "내 컴에선 되는데" 문제가 준다.

2부 · 4단계

main.py — 첫 FastAPI 앱

main.py — docpilot week01
from fastapi import FastAPI

app = FastAPI(title="docpilot", version="0.1.0")

@app.get("/")
def root() -> dict:
    return {"message": "Hello, DocPilot"}

@app.get("/health")
def health() -> dict:
    return {"status": "ok"}

GET /은 사람이 눈으로 보는 용도 · GET /health는 로드밸런서·쿠버네티스가 기계적으로 확인하는 용도. 4주차 이후 K8s probe가 이 엔드포인트를 부른다 — 지금부터 습관처럼 만든다.

2부 · 5단계

uvicorn으로 로컬 실행 (포트 바인딩)

bash — 서버 실행
uvicorn main:app --reload --host 127.0.0.1 --port 8000
main:app

main.py의 app 객체 실행

--reload

저장 시 자동 재시작(개발용)

--port 8000

8000번 포트로 노출 (12-Factor #7)

확인 — 이 로그가 뜨고 멈추지 않고 대기 
INFO:  Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:  Application startup complete.
2부 · 6단계 · 실행 확인

기대 출력이 나오는가

bash — 새 터미널에서 curl
curl http://127.0.0.1:8000/
curl http://127.0.0.1:8000/health
이 JSON이 나오면 성공 ✓ 
{"message":"Hello, DocPilot"}
{"status":"ok"}
/ → Hello, DocPilot
/health → status: ok
/docs → 자동 Swagger UI

확인이 끝나면 서버 터미널에서 Ctrl+C로 서버를 멈춘다.

확인: git status에 .venv/가 안 보여야 정상. git log --oneline에 방금 커밋 한 줄이 보인다.

3부 · 9단계

GitHub 저장소 생성 & 푸시

방법 A · gh CLI (권장)
bash
gh auth login
gh repo create docpilot --public \
  --source=. --remote=origin --push
방법 B · 웹에서 빈 저장소 후 연결
bash
git branch -M main
git remote add origin \
  https://github.com/<you>/docpilot.git
git push -u origin main
확인

저장소 페이지에 main.py · requirements.txt · .gitignore가 올라가 있으면 → 1주차 산출물 = 로컬 웹앱 + GitHub 저장소 완성!
