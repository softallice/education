개념 ① / 왜 오케스트레이터인가

Compose는 한 대, 실제 서비스는 그 이상

스케줄링

"여유 있는 노드에 이 컨테이너를 알아서 배치해줘."

자가치유

"죽으면 다시 띄워. 항상 N개가 떠 있게 유지해줘."

확장

"트래픽이 늘면 복제본을 늘리고, 줄면 줄여줘."

핵심 사고 전환 — 선언형(declarative)

K8s는 위 셋을 선언형으로 처리한다. 우리는 "결과가 이래야 한다(desired state)"만 YAML로 적고, 컨트롤러가 실제 상태를 그 선언에 계속 맞춘다(reconciliation loop). "이 명령을 실행해라(how)"가 아니라 "이렇게 돼 있어야 한다(what)"를 적는 것이 전환점이다.

개념 ② / 핵심 오브젝트

네 가지 오브젝트

오브젝트한 줄 정의docpilot 예
Pod배포의 최소 단위. 1개 이상의 컨테이너를 묶은 것. IP를 하나 가진다.uvicorn이 도는 docpilot 컨테이너 1개 = Pod 1개
ReplicaSet"동일한 Pod를 N개 유지"를 책임진다. 죽으면 다시 만든다.docpilot Pod 2개 유지
DeploymentReplicaSet을 관리하며 버전 롤아웃/롤백 담당. 우리가 직접 다루는 상위 오브젝트.docpilot Deployment
Service여러 Pod 앞에 고정 이름·IP를 주는 내부 로드밸런서. Pod IP가 바뀌어도 접속 지점은 그대로.docpilot Service (ClusterIP)
개념 ② / 관계도

Deployment가 Pod를 찍어내고, Service가 묶는다

Deployment · 버전 · 롤아웃 관리
ReplicaSet · 개수 유지 replicas=2

docpilot 컨테이너 · IP 수시 변동
Pod
docpilot 컨테이너 · IP 수시 변동
▲   label selector: app: docpilot   ▲
Service · docpilot
ClusterIP · 고정 이름/IP
selector로 위 Pod들에 부하 분산

왜 Service가 필요한가? Pod는 언제든 죽고 다시 생기며 그때마다 IP가 바뀐다. 그래서 Pod IP로 직접 접속하면 안 된다. Service는 label selector(예: app: docpilot)로 대상 Pod들을 묶어 바뀌지 않는 접속 지점을 제공한다.

개념 ③ / kubectl

1부 · 1단계

k8s/ 디렉터리 + 클러스터 설정 파일

k8s/kind-cluster.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
name: docpilot
nodes:
  - role: control-plane
    kubeadmConfigPatches:
      - |
        kind: InitConfiguration
        nodeRegistration:
          kubeletExtraArgs:
            node-labels: "ingress-ready=true"
    extraPortMappings:
      - containerPort: 80
        hostPort: 80
        protocol: TCP
      - containerPort: 443
        hostPort: 443
        protocol: TCP
bash
mkdir -p k8s

노드 1개짜리 단순 클러스터지만, 5주차 Ingress에 대비해 호스트 80/443 포트를 노드로 매핑하고 ingress-ready 라벨을 미리 붙인다.

이 포트 매핑은 클러스터 생성 시점에만 지정할 수 있어 지금 넣어 둔다.

docker build -t docpilot:0.3.0 .

왜 새 태그인가

week02에서 이미지를 docpilot:0.2.0으로 만들었고, week03에서 /documents가 추가됐다. 현재 코드 상태(앱 version="0.3.0")에 맞춰 docpilot:0.3.0 새 태그로 빌드한다.

기억할 차이

kind 클러스터는 로컬 Docker 저장소와 분리된 별도 환경이다. 빌드만 해선 클러스터가 이미지를 못 본다 → 다음 단계에서 kind load가 필요하다.

2부 · 4단계 · 절대 누락 금지

kind load로 이미지를 클러스터에 주입

bash — 로컬 이미지를 노드 안으로 복사
kind load docker-image docpilot:0.3.0 \
  --name docpilot
# Image: "docpilot:0.3.0" ... loaded

docker exec -it docpilot-control-plane \
  crictl images | grep docpilot
이 단계를 빼먹으면?

뒤에서 ImagePullBackOff가 난다. K8s가 레지스트리에서 docpilot:0.3.0을 받아오려다 실패하기 때문이다. 그래서 Deployment에 imagePullPolicy: IfNotPresent를 명시해 "로컬에 있으면 받아오지 마"라고 지시할 것이다.

postgres:16-alpine는 공개 이미지라 클러스터가 인터넷에서 자동으로 받아온다. 로컬 빌드 이미지(docpilot)만 kind load가 필요하다.

3부 · 5단계

PostgreSQL 배포 — docpilot의 의존 DB

k8s/postgres.yaml — Deployment (발췌)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: docpilot-db
  labels: { app: docpilot-db }
spec:
  replicas: 1
  selector: { matchLabels: { app: docpilot-db } }
  template:
    metadata: { labels: { app: docpilot-db } }
    spec:
      containers:
        - name: postgres
          image: postgres:16-alpine
          ports: [{ containerPort: 5432 }]
          env:            # week05에서 Secret으로 분리
            - { name: POSTGRES_USER,     value: "docpilot" }
            - { name: POSTGRES_PASSWORD, value: "docpilot" }
            - { name: POSTGRES_DB,       value: "docpilot" }
          volumeMounts: [{ name: data, mountPath: /var/lib/postgresql/data }]
      volumes: [{ name: data, emptyDir: {} }]   # 실습용: Pod 소멸 시 소실
k8s/postgres.yaml — Service
apiVersion: v1
kind: Service
metadata: { name: docpilot-db }
spec:
  selector: { app: docpilot-db }
  ports: [{ port: 5432, targetPort: 5432 }]
  type: ClusterIP
bash
kubectl apply -f k8s/postgres.yaml
kubectl get pods -l app=docpilot-db  # 1/1 Running 대기

Service 이름 docpilot-db가 곧 docpilot의 접속 호스트명이 된다(클러스터 내부 DNS).

3부 · 6단계

deployment.yaml — docpilot replicas=2

k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: docpilot, labels: { app: docpilot } }
spec:
  replicas: 2                 # Pod 2개를 항상 유지
  selector: { matchLabels: { app: docpilot } }
  template:                   # 이 틀로 Pod를 찍어낸다
    metadata: { labels: { app: docpilot } }   # Service가 이 라벨로 Pod를 찾음
    spec:
      containers:
        - name: docpilot
          image: docpilot:0.3.0
          imagePullPolicy: IfNotPresent   # 로컬 로드 이미지 사용(ImagePullBackOff 예방)
          ports: [{ containerPort: 8000 }]
          env:            # 아직 하드코딩. week05에서 ConfigMap/Secret으로 외부화
            - { name: APP_ENV, value: "k8s-dev" }
            - name: DATABASE_URL   # 호스트는 Service명 docpilot-db
              value: "postgresql+psycopg://docpilot:docpilot@docpilot-db:5432/docpilot"

week03 compose의 db 대신 K8s Service명 docpilot-db가 호스트다. psycopg 스킴은 week03과 동일하다.

3부 · 7단계

service.yaml — Pod 앞의 고정 접속 지점

k8s/service.yaml
apiVersion: v1
kind: Service
metadata: { name: docpilot }
spec:
  selector: { app: docpilot }   # 이 라벨 Pod로 트래픽 분산
  ports:
    - port: 80              # Service가 노출하는 포트
      targetPort: 8000      # 컨테이너(uvicorn) 포트
  type: ClusterIP           # 클러스터 내부 전용(기본)
포트 매핑 읽기
  • ·port: 80 — Service가 여는 문
  • ·targetPort: 8000 — 실제 컨테이너 포트
  • ·selector: app=docpilot — 이 라벨 Pod에 부하 분산

ClusterIP는 클러스터 내부 전용이라 밖에서 바로 못 붙는다. 개발 중 확인엔 port-forward를 쓴다(10단계).

4부 · 11단계 · 스케일

replicas를 늘리고 줄이기

bash — 명령형 스케일
kubectl scale deployment/docpilot --replicas=4
kubectl get pods -l app=docpilot   # 4개로 증가

kubectl scale deployment/docpilot --replicas=2
kubectl get pods -l app=docpilot   # 2개로 감소
# 초과분이 Terminating
선언형이 진실의 원천

방금은 명령형으로 조정했다. 선언형으로 하려면 deployment.yaml의 replicas 값을 고치고 다시 kubectl apply한다.

실무에선 파일이 source of truth여야 하므로 파일을 고쳐 apply하는 습관을 들이자.

4부 · 12단계 · 이번 주 하이라이트

Pod를 지워도 스스로 살아난다

bash — 감시 창(별도 터미널)
kubectl get pods -l app=docpilot -w
bash — 다른 터미널에서 Pod 삭제
POD=$(kubectl get pod -l app=docpilot \
  -o jsonpath='{.items[0].metadata.name}')
kubectl delete pod "$POD"
감시 창에서 보이는 것 ✓ 
docpilot-...-abcde  Terminating
docpilot-...-klmno  ContainerCreating
docpilot-...-klmno  Running

지운 Pod가 Terminating 되는 동시에 새 Pod가 곧바로 생성된다. 잠시 뒤에도 항상 2개. 이것이 "desired state = 2개"를 K8s가 지켜주는 자가치유다.

(선택) 무중단 감 잡기: port-forward를 켜고 반복 curl 중 Pod를 지우면 대부분 200. 간헐적 실패는 아직 readiness probe가 없어서다 — week05에서 probe로 해결한다.
