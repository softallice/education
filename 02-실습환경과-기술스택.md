## 1. 단계별 설치 가이드

각 도구마다 **설치 명령**과 **확인 명령**(버전 출력)을 함께 제시한다. OS별로 방법이 다르면 Windows/WSL2 · macOS · Linux로 나눠 적었다.

> **Windows 사용자 필독**: 이 강의의 컨테이너/쿠버네티스 실습은 **WSL2(Ubuntu) 위에서** 진행하는 것을 강력히 권장한다. Windows 네이티브 PowerShell이 아니라, 아래 WSL2를 먼저 설치하고 그 안의 Ubuntu 터미널에서 Linux 명령을 따라 하면 macOS/Linux와 동일한 경험을 얻는다.

### 1-0. (Windows 전용) WSL2 설치

무엇을/왜: Windows에서 Linux 개발 환경을 그대로 쓰기 위해 WSL2 + Ubuntu를 설치한다.

```powershell
# PowerShell을 "관리자 권한"으로 실행 후:
wsl --install -d Ubuntu-24.04
# 설치 후 재부팅. 재부팅 뒤 Ubuntu 초기 사용자/암호 설정.
```

확인: PowerShell에서 아래를 실행해 `VERSION`이 `2`로 나오면 성공.

```powershell
wsl --status
wsl -l -v
# 기대 출력 예: NAME  STATE  VERSION  →  Ubuntu-24.04  Running  2
```

이후 모든 Linux 명령은 **Ubuntu(WSL2) 터미널**에서 실행한다. (시작 메뉴 → "Ubuntu" 실행)

### 1-1. Python 3.12 (+ uv 또는 venv)

무엇을/왜: `docpilot`을 포함한 모든 실습의 기본 언어. 버전 혼선을 막기 위해 **3.12**로 고정한다.

**Windows/WSL2 · Linux (Ubuntu/Debian)**
```bash
# deadsnakes PPA로 Python 3.12 설치 (Ubuntu 22.04/24.04)
sudo apt update
sudo apt install -y software-properties-common
sudo add-apt-repository -y ppa:deadsnakes/ppa
sudo apt update
sudo apt install -y python3.12 python3.12-venv python3-pip
```

**macOS (Homebrew)**
```bash
# Homebrew가 없으면 먼저 설치: https://brew.sh
brew install python@3.12
```

확인:
```bash
python3.12 --version
# 기대 출력: Python 3.12.x
```

**의존성 관리 — `uv` 권장 (빠름) 또는 `venv`+`pip`**

`uv`는 초고속 Python 패키지·가상환경 관리자다. 이 강의는 `uv`를 표준으로 삼되, 익숙하면 `venv`를 써도 된다.

```bash
# 방법 A: uv 설치 (macOS/Linux/WSL2 공통)
curl -LsSf https://astral.sh/uv/install.sh | sh
# 새 셸을 열거나 아래로 PATH 반영
source "$HOME/.local/bin/env" 2>/dev/null || export PATH="$HOME/.local/bin:$PATH"
```

확인:
```bash
uv --version
# 기대 출력: uv 0.x.x
```

`uv`로 프로젝트 가상환경 만들기(3장에서 실제로 사용):
```bash
uv venv --python 3.12        # .venv/ 생성
source .venv/bin/activate     # 활성화 (Windows PowerShell: .venv\Scripts\Activate.ps1)
```

방법 B: 표준 `venv`만 쓸 경우
```bash
python3.12 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
```

확인(가상환경 활성화 상태에서):
```bash
python --version   # Python 3.12.x
which python       # .../.venv/bin/python  (가상환경 경로여야 함)
```

### 1-3. Docker Desktop / Docker Engine

무엇을/왜: Week 2부터 `docpilot`을 컨테이너로 빌드·실행한다. 컨테이너 실습의 핵심 도구.

**Windows/WSL2**: Docker Desktop 설치 후 **WSL2 integration**을 켠다.
- https://www.docker.com/products/docker-desktop 에서 Docker Desktop 설치.
- Docker Desktop → Settings → Resources → WSL Integration → 사용 중인 Ubuntu 배포판 토글 ON.

**macOS**: Docker Desktop 설치(Apple Silicon/Intel 각각의 dmg).
```bash
brew install --cask docker   # 설치 후 Docker Desktop 앱을 한 번 실행해 데몬 기동
```

**Linux (Ubuntu, Docker Engine)**: Desktop 없이 엔진만 설치.
```bash
# 공식 편의 스크립트
curl -fsSL https://get.docker.com | sudo sh
# sudo 없이 docker 쓰려면 현재 사용자를 docker 그룹에 추가 후 재로그인
sudo usermod -aG docker $USER
newgrp docker
```

확인 1 (버전):
```bash
docker --version
docker compose version
# 기대 출력: Docker version 2x.x.x / Docker Compose version v2.x.x
```

확인 2 (실제 컨테이너 실행 — 반드시 통과할 것):
```bash
docker run hello-world
# 기대 출력에 다음 문장이 포함되어야 함:
# "Hello from Docker!"
# "This message shows that your installation appears to be working correctly."
```

> `permission denied ... /var/run/docker.sock` 오류가 나면 위의 `usermod -aG docker $USER` 후 터미널을 새로 열거나 로그아웃/로그인한다. Docker Desktop 사용자는 앱이 실행 중인지 확인한다.

### 1-4. kind (기본) + kubectl

무엇을/왜: Week 4~5에서 `docpilot`을 로컬 Kubernetes 클러스터에 배포한다. **kind**(Kubernetes IN Docker)는 Docker 컨테이너로 K8s 클러스터를 띄우는 가장 가벼운 방법이라 기본으로 채택한다. Docker가 먼저 동작해야 한다.

**kubectl 설치 (K8s를 조작하는 CLI)**

Linux/WSL2:
```bash
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl
rm kubectl
```

macOS:
```bash
brew install kubectl
```

확인:
```bash
kubectl version --client
# 기대 출력: Client Version: v1.3x.x
```

**kind 설치**

Linux/WSL2:
```bash
# amd64 기준. Apple Silicon/arm은 -amd64 를 -arm64 로 교체.
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.24.0/kind-linux-amd64
chmod +x ./kind
sudo mv ./kind /usr/local/bin/kind
```

macOS:
```bash
brew install kind
```

확인:
```bash
kind --version
# 기대 출력: kind version 0.24.0
```

**확인(핵심) — 실제 클러스터 생성 및 노드 조회**
```bash
# 1) 클러스터 생성 (Docker 위에 K8s 노드 컨테이너가 뜬다)
kind create cluster --name docpilot
# 2) 노드 확인
kubectl get nodes
# 기대 출력 예:
# NAME                     STATUS   ROLES           AGE   VERSION
# docpilot-control-plane   Ready    control-plane   60s   v1.3x.x
```

실습이 끝나면 리소스를 정리한다(디스크/메모리 절약):
```bash
kind delete cluster --name docpilot
```

**대안: minikube**

kind가 환경에 맞지 않으면 minikube를 써도 된다(수업은 kind 기준으로 진행).
```bash
# Linux
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube
# macOS
brew install minikube

minikube start                 # 클러스터 시작
kubectl get nodes              # Ready 확인
minikube stop                  # 정리
```

## 2. AI API 키 발급·관리

Week 6부터 LLM/임베딩 API를 호출한다. 키 발급 경로와 **안전한 관리 원칙**을 미리 익혀 둔다.

### 2-1. 키 발급 경로

| 제공자 | 발급 위치 | 환경변수 이름(이 강의 표준) |
|---|---|---|
| OpenAI | https://platform.openai.com → API keys | `OPENAI_API_KEY` |
| Google Gemini | https://aistudio.google.com/app/apikey (Google AI Studio) | `GEMINI_API_KEY` |
| Hugging Face | https://huggingface.co/settings/tokens (Access Tokens, `read` 권한) | `HUGGINGFACE_API_KEY` |

### 2-2. `.env` + 환경변수로 관리 (하드코딩 절대 금지)

무엇을/왜: **API 키를 소스코드에 직접 적으면 안 된다.** 실수로 GitHub에 올라가면 키가 유출되어 요금 폭탄·계정 정지로 이어진다. 반드시 `.env` 파일 + 환경변수로 분리한다.

프로젝트 루트에 `.env` 파일을 만든다:
```bash
# .env  (이 파일은 절대 커밋하지 않는다)
OPENAI_API_KEY=sk-...여기에_본인_키...
GEMINI_API_KEY=...여기에_본인_키...
HUGGINGFACE_API_KEY=hf_...여기에_본인_키...
```

Python에서 불러오는 표준 방법(`python-dotenv` 사용):
```python
# config.py — 키를 코드에 적지 않고 환경에서 읽어온다
import os
from dotenv import load_dotenv

load_dotenv()  # 프로젝트 루트의 .env 를 환경변수로 로드

OPENAI_API_KEY = os.environ["OPENAI_API_KEY"]  # 없으면 KeyError로 즉시 실패 → 조기 발견
GEMINI_API_KEY = os.environ.get("GEMINI_API_KEY")  # 선택 항목은 get() 사용

# 금지 예시 (절대 이렇게 하지 말 것):
#   client = OpenAI(api_key="sk-abcd1234...")   # 하드코딩 → 유출 위험
```

### 2-3. `.gitignore`에 `.env` 추가 (필수)

무엇을/왜: `.env`가 실수로라도 Git에 올라가지 않게 처음부터 무시 목록에 넣는다.

```bash
# .gitignore
.env
.env.*
!.env.example      # 예시 템플릿은 커밋 허용
.venv/
__pycache__/
*.pyc
```

팀원이 어떤 키가 필요한지 알도록 **값이 비어 있는** `.env.example`은 커밋한다:
```bash
# .env.example  (실제 값 없이 키 이름만)
OPENAI_API_KEY=
GEMINI_API_KEY=
HUGGINGFACE_API_KEY=
```

확인(키가 코드/커밋에 노출되지 않았는지 점검):
```bash
git status --ignored | grep .env
# 기대: .env 가 "Ignored files" 목록에 있어야 함 (추적 대상이 아님)
git ls-files | grep -E "\.env$" && echo "경고: .env가 추적됨!" || echo "OK: .env 미추적"
```

## 3. docpilot 프로젝트 초기 스캐폴드

무엇을/왜: 12주 내내 자라날 `docpilot` 프로젝트의 뼈대를 미리 만들어 둔다. Week 1에서 이 구조 위에 FastAPI `GET /`를 추가한다.

### 3-1. 디렉토리 구조

```text
docpilot/
├── app/
│   ├── __init__.py
│   ├── main.py              # FastAPI 진입점 (Week 1에서 작성)
│   ├── config.py            # .env 로드·설정 (2장 참고)
│   └── routers/             # 기능별 라우터 (주차마다 추가: chat, ask, agent...)
│       └── __init__.py
├── tests/
│   └── test_health.py       # 헬스체크 테스트 (Week 1)
├── .env                     # 실제 키 (커밋 금지)
├── .env.example             # 키 템플릿 (커밋)
├── .gitignore
├── requirements.txt         # (또는 pyproject.toml)
├── Dockerfile               # Week 2에서 추가
```

## 4. 비용 관리 (유료 API 부담 줄이기)

유료 LLM API는 학생에게 부담이 될 수 있다. 아래 대안을 적극 활용한다.

- **Hugging Face 무료 티어**: HF Inference API/Hub는 무료 사용 한도가 있어 임베딩·소형 모델 실습에 충분하다. Week 7~8의 상당 부분을 HF로 대체할 수 있다.
- **로컬 모델 (Ollama)**: 인터넷/요금 없이 노트북에서 소형 LLM을 돌린다. 유료 API 대체 실습 경로로 준비한다.
  ```bash
  # 설치 (macOS/Linux/WSL2)
  curl -fsSL https://ollama.com/install.sh | sh
  ollama pull llama3.2:1b     # 가벼운 모델 받기
  ollama run llama3.2:1b "Hello"   # 로컬 추론 확인
  ```
  확인: `ollama --version` → `ollama version ...` 출력.
- **학과 공용 키 + 사용량 제한**: 개인 결제가 어려우면 담당 교수/조교가 발급한 공용 키를 사용량 상한과 함께 배포하는 방식을 이용한다(수업 공지 확인). 공용 키도 절대 코드에 하드코딩하지 않는다.
- **클라우드 크레딧(교육 프로그램)**: 배포 실습·크레딧이 필요하면 아래 학생 프로그램을 활용한다.
  - GitHub Student Developer Pack (https://education.github.com/pack)
  - Google Cloud / AWS Educate / Azure for Students 등 주요 CSP의 학생 무료 크레딧
  - OpenAI/Gemini 신규 가입 무료 크레딧(제공 시)

> 원칙: **먼저 무료/로컬 대안으로 개발·디버깅**하고, 최종 확인 때만 유료 API를 호출해 토큰 낭비를 줄인다.

## 6. 트러블슈팅 (흔한 설치 오류)

| 증상 | 원인 | 해결 |
|---|---|---|
| `wsl --install` 후에도 VERSION 1 | 구버전 WSL / 가상화 미활성 | BIOS에서 가상화(VT-x/AMD-V) 켜기, `wsl --set-default-version 2`, `wsl --update` |
| `python3.12: command not found` | PPA 미추가 또는 PATH 문제 | `add-apt-repository ppa:deadsnakes/ppa` 재실행, macOS는 `brew link python@3.12` |
| `docker: permission denied ... docker.sock` | 사용자가 docker 그룹에 없음 | `sudo usermod -aG docker $USER` 후 재로그인 / `newgrp docker` |
| `docker run hello-world` 멈춤·오류 | Docker 데몬 미기동 | Docker Desktop 실행 확인, Linux는 `sudo systemctl start docker` |
| Windows에서 `docker` 명령 없음 | WSL Integration 미설정 | Docker Desktop → Settings → Resources → WSL Integration 토글 ON |
| `kind create cluster` 실패 | Docker 미기동 / 메모리 부족 | Docker 먼저 확인, Docker Desktop 리소스(RAM 4GB+) 상향 |
| `kubectl` 이 클러스터에 연결 안 됨 | context 미설정 | `kubectl config get-contexts`, `kubectl config use-context kind-docpilot` |
| `kind`/`kubectl` 아키텍처 오류(arm) | Apple Silicon에서 amd64 바이너리 | 다운로드 URL을 `-arm64`로 교체, 또는 `brew install`로 설치 |
| `ModuleNotFoundError: fastapi` | 가상환경 미활성 / 설치 누락 | `source .venv/bin/activate` 후 `pip install -r requirements.txt` |
| `KeyError: 'OPENAI_API_KEY'` | `.env` 미로드 / 오타 | `load_dotenv()` 호출 확인, `.env` 위치가 실행 디렉토리 기준인지 확인 |
| API 호출 시 401 Unauthorized | 키 오류/만료/공백 | 키 재발급, `.env` 값 앞뒤 공백·따옴표 제거 |
| API 429 Too Many Requests | 무료 한도 초과 | 잠시 대기, Ollama 로컬 모델/HF 무료 티어로 전환 |
| `ssh -T git@github.com` 실패 | SSH 키 미등록 | `cat ~/.ssh/id_ed25519.pub` 내용을 GitHub SSH keys에 등록 |
